<task type="auto">
  <name>Task 1: Create the sequential runner orchestration module</name>
  <files>src/lib/autonomous/runner.ts</files>
  <action>
Create `src/lib/autonomous/runner.ts` that exports `runAutonomousLoop()` and the `RunResult` interface.

1. `RunResult` interface:
   - `agentId: string`
   - `agentName: string`
   - `serviceType: string`
   - `actions: { action: string; status: 'success' | 'failure'; details: Record<string, unknown> }[]`
   - `logFilecoinCid: string | null`

2. `validateEnvironment()` (not exported):
   - Check for required env vars: FILECOIN_PRIVATE_KEY, AGENT_PAYMENT_ADDRESS
   - Throw descriptive error listing which vars are missing
   - Call this at the top of runAutonomousLoop() before any DB queries

3. `runAutonomousLoop()`:
   - Call validateEnvironment()
   - Query all agents from DB: `SELECT * FROM agents`
   - Initialize empty results array
   - For each agent (sequentially, NOT parallel):
     a. `buildAgentLog(agent)` to create fresh log
     b. Get scenario: `AGENT_SCENARIOS[agent.service_type || 'general']`
     c. Step 1: `registerIdentityAction(agent, log)` -- idempotent
     d. Step 2: `createBountyAction(agent, scenario, log)` -- creates bounty for other agents
     e. Step 3: `discoverAndClaimBounty(agent, log)` -- find and claim matching bounty
     f. Step 4: For first post in scenario.posts, call `createPostAction(agent, content, log)`
     g. Step 5: `mintPostNFTAction(agent, postId, log)` -- mint that post as NFT
     h. Step 6: If bountyId from step 3, call `completeBountyAction(agent, bountyId, log)`
     i. Step 7: `uploadLogAction(log)` -- upload final log to Filecoin
     j. Build RunResult from log entries and push to results
   - Wrap the entire per-agent block in try/catch so one agent failure does not abort the loop
   - On agent-level failure, still attempt uploadLogAction with partial log, then push result with whatever was collected
   - Return results array

4. Store last run results in a module-level variable `let lastRunResults: RunResult[] | null = null` and export `getLastRunResults()` for the status endpoint.

Import from `@/lib/autonomous/agent-actions` and `@/lib/autonomous/demo-scenarios`. Import `getDb` from `@/lib/db`. Import `buildAgentLog` from `@/lib/agent-log`. Import `type Agent` from `@/lib/types`.

IMPORTANT: Re-read the agent from DB after registerIdentityAction completes (it may have updated erc8004_token_id). Use fresh agent data for subsequent steps.

Create two API route files:

**POST /api/autonomous/run** (`src/app/api/autonomous/run/route.ts`):
- Import `runAutonomousLoop` from `@/lib/autonomous/runner`
- Call `runAutonomousLoop()` and return `Response.json({ status: 'complete', agentCount: results.length, results })`
- On error, return `Response.json({ error: message }, { status: 500 })`
- This is a long-running request (could take several minutes). No timeout handling needed for demo purposes.

**GET /api/autonomous/status** (`src/app/api/autonomous/status/route.ts`):
- Import `getLastRunResults` from `@/lib/autonomous/runner`
- Return the last run results: `Response.json({ hasRun: results !== null, results })`
- If no run has happened yet, return `{ hasRun: false, results: null }`
- Set `Cache-Control: no-cache` header so polling always gets fresh data

Both routes should be minimal -- all logic lives in the runner module.
