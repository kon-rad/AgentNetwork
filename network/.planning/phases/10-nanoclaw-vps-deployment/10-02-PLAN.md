  <action>
    Test whether Railway containers support WireGuard by running a temporary Railway deploy with a WireGuard client configured.

    **Step 1 — Generate WireGuard keypairs:**
    On your local machine:
    ```bash
    # VPS peer keypair
    wg genkey | tee vps_private.key | wg pubkey > vps_public.key
    # Railway peer keypair
    wg genkey | tee railway_private.key | wg pubkey > railway_public.key
    cat vps_private.key vps_public.key railway_private.key railway_public.key
    ```
    Store these keys temporarily — do NOT commit them to git.

    **Step 2 — Configure WireGuard on VPS:**
    ```bash
    ssh deploy@VPS_IP sudo bash -c "
    cat > /etc/wireguard/wg0.conf << 'EOF'
    [Interface]
    Address = 10.0.0.1/24
    PrivateKey = VPS_PRIVATE_KEY
    ListenPort = 51820

    [Peer]
    PublicKey = RAILWAY_PUBLIC_KEY
    AllowedIPs = 10.0.0.2/32
    EOF
    systemctl enable wg-quick@wg0
    wg-quick up wg0
    wg show
    "
    ```

    **Step 3 — Test WireGuard from Railway:**
    Create a temporary test script `agent-server/scripts/test-wireguard.sh`:
    ```bash
    #!/bin/bash
    # Install wireguard-tools
    apt-get update -qq && apt-get install -y -qq wireguard-tools iproute2

    # Configure Railway peer
    cat > /etc/wireguard/wg0.conf << 'EOF'
    [Interface]
    Address = 10.0.0.2/24
    PrivateKey = RAILWAY_PRIVATE_KEY

    [Peer]
    PublicKey = VPS_PUBLIC_KEY
    Endpoint = VPS_IP:51820
    AllowedIPs = 10.0.0.1/32
    PersistentKeepalive = 25
    EOF

    # Attempt to bring up WireGuard interface
    wg-quick up wg0 2>&1
    WG_EXIT=$?

    if [ $WG_EXIT -eq 0 ]; then
      echo "RESULT: WireGuard UP successfully"
      ping -c 3 10.0.0.1 && echo "RESULT: Ping to VPS succeeded" || echo "RESULT: Ping failed (UDP may be blocked)"
    else
      echo "RESULT: wg-quick failed (likely NET_ADMIN not granted)"
    fi
    ```

    Deploy this as a one-off Railway job or add to an existing Railway service start command temporarily. Check Railway logs for the RESULT lines.

    **Alternatively (faster spike):** If Railway CLI is available:
    ```bash
    railway run bash agent-server/scripts/test-wireguard.sh
    ```

    **Step 4 — Document outcome in TRANSPORT.md:**

    Create `.planning/phases/10-nanoclaw-vps-deployment/TRANSPORT.md`:

    If WireGuard succeeded:
    ```markdown
    # Transport Decision: WireGuard Confirmed

    **Result:** WireGuard UP succeeded in Railway container. NET_ADMIN is available.
    **Transport:** WireGuard tunnel (10.0.0.x) — Railway is peer 10.0.0.2, VPS is peer 10.0.0.1
    **Implementation:** Plans 10-03 and 10-04 use WireGuard private IP for NanoClaw calls
    **VPS WireGuard IP:** 10.0.0.1
    **NanoClaw URL from Railway:** http://10.0.0.1:3000 (no Caddy needed for encryption)
    ```

    If WireGuard failed (expected):
    ```markdown
    # Transport Decision: HTTPS + Caddy (WireGuard not viable on Railway)

    **Result:** wg-quick returned RTNETLINK Operation not permitted — NET_ADMIN not granted on Railway
    **Transport:** HTTPS + Caddy on VPS with shared secret header (x-shared-secret)
    **Security:** TLS 1.3 via Caddy auto-HTTPS (Let's Encrypt) + shared secret header for auth
    **NanoClaw URL from Railway:** https://nanoclaw.DOMAIN (Caddy terminates TLS, reverse-proxies to localhost:3000)
    **Domain needed:** Requires DNS A record pointing to VPS IP (configure in plan 10-04)
    ```

    **Cleanup:** Delete the temporary WireGuard config from VPS if WireGuard failed. Delete test script. Do NOT commit private keys.
