<task type="auto">
  <name>Task 2: Create Providers component and update layout.tsx for SSR</name>
  <files>
    src/components/layout/providers.tsx
    src/app/layout.tsx
    next.config.ts
  </files>
  <action>
1. Create `src/components/layout/providers.tsx` as a `'use client'` component:
   - Import `QueryClient`, `QueryClientProvider` from `@tanstack/react-query`.
   - Import `RainbowKitProvider` from `@rainbow-me/rainbowkit`.
   - Import `WagmiProvider`, `cookieToInitialState` from `wagmi`.
   - Import `wagmiConfig` from `@/lib/wagmi`.
   - Import `@rainbow-me/rainbowkit/styles.css` (import the stylesheet HERE in the client component, NOT in globals.css — prevents CSS conflict with Tailwind 4).
   - Accept props: `{ children: React.ReactNode; cookie?: string | null }`.
   - Use `useState` to create a stable `QueryClient` instance.
   - Call `cookieToInitialState(wagmiConfig, cookie)` to derive initial state.
   - Render: `WagmiProvider` (with config + initialState) wrapping `QueryClientProvider` wrapping `RainbowKitProvider` wrapping `{children}`.

2. Update `src/app/layout.tsx`:
   - CRITICAL: Make the default export an `async` function (required for Next.js 16 async Request APIs).
   - Import `cookies` from `next/headers`.
   - Import `Providers` from `@/components/layout/providers`.
   - Inside the function body: `const cookieStore = await cookies()` — MUST await (synchronous cookies() throws in Next.js 16).
   - Get cookie string: `const cookie = cookieStore.toString()`.
   - Wrap `{children}` with `<Providers cookie={cookie}>`.
   - Keep the existing `Navbar` inside the Providers wrapper (Navbar will need wagmi hooks in Plan 02).
   - Keep existing font loading (Geist Sans + Geist Mono) — font fix happens in Plan 03.
   - Keep existing className on html and body tags.

3. Check `next.config.ts` — do NOT add any webpack config. The current empty config is correct for Turbopack default in Next.js 16. wagmi, viem, and RainbowKit are ESM-native and do not require Node.js polyfills (crypto, buffer, stream, etc.) — Turbopack handles them natively without any resolveAlias or polyfill configuration. Only add `turbopack.resolveAlias` if the build fails (it should not).

4. Run `pnpm build` to verify the build passes with Turbopack. If it fails due to the placeholder WalletConnect project ID, that is acceptable — the important thing is no module resolution or polyfill errors. If there are polyfill errors (e.g., "Can't resolve 'crypto'"), add the necessary entries to `turbopack.resolveAlias` in next.config.ts.
  </action>
  <verify>
Run `pnpm build 2>&1 | tail -20` — build should complete (or fail only due to missing env var, not due to module/polyfill errors).
Run `grep -c "use client" src/components/layout/providers.tsx` — should return 1.
Run `grep "await cookies" src/app/layout.tsx` — should match, confirming async cookie read.
  </verify>
  <done>Providers.tsx wraps the app with WagmiProvider + QueryClientProvider + RainbowKitProvider. layout.tsx is async and reads cookies for SSR hydration. Build passes with Turbopack (no polyfill errors).</done>
